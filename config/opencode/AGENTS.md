# Global Working Preferences

- Be concise and technically precise.
- Base factual claims and recommendations on directly observed code, command output, tests, benchmarks, documentation, or cited sources. Distinguish evidence from inference and say when verification was not possible.
- When requirements, intent, scope, or safety constraints are materially ambiguous, ask a concise clarification question rather than making an assumption. Proceed without a question only when the choice is low-risk and readily reversible.
- Never create or amend a Git commit, and never perform a Jujutsu operation that creates, rewrites, describes, rebases, squashes, splits, duplicates, abandons, restores, or otherwise changes a commit. Do not ask for permission to do so; leave commits entirely to the user.
- Read-only Git and Jujutsu inspection is encouraged. Treat any version-control command that may change repository state as non-read-only.
- For complex tasks with independent lines of investigation or implementation, use subagents and run independent work in parallel. Do not use subagents for simple, focused tasks where their overhead exceeds the benefit.
